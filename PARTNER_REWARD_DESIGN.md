# METI 제휴·리워드 설계서 (해피트리 연동 / 글로벌)

> 작성: 2026-06-24 · 대상: 앱팀 + 서버팀(THE-METI) 공유
> 결정 배경: 제휴 탭을 앱테크(리워드) 허브로. 글로벌 출시 전제 → 인앱결제(IAP) 정책이 설계를 좌우.

---

## 0. 한 줄 원칙

> **"파는 건 웹(별도 법인), 앱은 동작만, 결제 유도는 앱 밖에서."**

글로벌에선 앱 안에서 디지털 게임재화(별/하트)를 팔거나 외부결제로 **유도(링크·버튼·문구)** 하면 Apple/Google IAP 강제 또는 앱 리젝. 이를 원천 회피하는 **오퍼월(리워드 애그리게이터) 모델**을 채택한다. (= 기획서 v1.7 §15 S2S 설계 방향)

---

## 1. 법인/표면 분리

| 주체 | 형태 | 역할 |
|------|------|------|
| **METI** (법인 A) | iOS/Android 앱 + 웹 | 명함·커뮤니티 + **제휴/리워드 허브**. METI 포인트 운영 |
| **해피트리** (법인 B) | **웹 우선 사이트** (별도 도메인) | 게임 본체. 별·하트·코인 경제 운영. **별 결제는 해피트리 웹에서** |

- 해피트리는 **웹 우선**으로 만들어 별 결제를 100% 해피트리 웹(Stripe 등 PG)에서 처리 → 앱마켓 30% 무관.
- METI 앱에서 해피트리는 **외부 브라우저로 열기**(인앱 WebView 결제 ❌ — METI 리젝 위험).

---

## 2. 화폐/재화 구조

| 재화 | 소유 | 획득 방법 | 결제 위치 | IAP |
|------|------|----------|----------|-----|
| **METI 포인트** | METI | **적립 전용**(게임·미션·출석). 직접 판매 ❌ | — | 원천 회피 ✅ |
| 🌟 **별** (대형 매출) | 해피트리 | 구매 | **해피트리 웹**(Stripe) | METI 무관 ✅ |
| ❤️ **하트** | 해피트리 | METI 포인트로 교환 / 게임 플레이 | 가상거래(포인트 차감) | 안 걸림 ✅ |
| 🪙 코인 | 해피트리 | 게임 레벨업 보상 | — | — |
| **METI 구독**(pro/biz) | METI | 구매 | **앱 IAP** / 웹 | IAP 표준 ✅ |

> 핵심 발상: 유저에게 화폐를 **팔지 않고 벌게** 한다. 돈은 **유저가 아니라 파트너(광고주)가** METI에 낸다(B2B). → IAP 무관.

## 3. 데이터/정산 흐름

```
┌────────────── METI 앱 (법인 A) ──────────────┐
│  제휴/리워드 탭                                │
│   · METI 포인트 잔액(적립 전용)                │
│   · 게임/미션 목록 · 적립 내역                 │
│   · 해피트리 = "외부 브라우저로 열기"          │
└───────────────┬───────────────────────────────┘
                │ (1) 송객 (deep link / 외부 브라우저)
                ▼
┌────────────── 해피트리 웹 (법인 B) ───────────┐
│  게임 본체 · 별 구매(Stripe) · 하트/코인       │
└───────────────┬───────────────────────────────┘
                │ (2) S2S 이벤트 (서버↔서버)
                │   "userX가 별 구매/액션 완료"
                ▼
┌────────────── METI 백엔드 (THE-METI) ─────────┐
│  POST /partner/rewards  → METI 포인트 적립     │
│  + 정산 카운트(송객/CPA) 기록                  │
└───────────────┬───────────────────────────────┘
                │ (3) 월정산 (B2B 인보이스)
                ▼
        해피트리(법인 B) → METI(법인 A) 레브셰어 지급
        ※ B2B 거래 → IAP 무관
```

- **(1) 송객**: METI 앱 → 해피트리 웹 (외부 브라우저). 유저 식별을 위해 매핑 토큰 전달 → `POST /partner/users/map`.
- **(2) S2S 적립**: 해피트리가 유저 행동(별 구매·미션 완료)을 METI 서버로 통지 → `POST /partner/rewards` → METI 포인트 적립 + 정산 데이터 누적.
- **(3) 정산**: 누적된 송객/CPA/레브셰어를 월단위 B2B로 해피트리 → METI 지급.

## 4. IAP 컴플라이언스 규칙 (글로벌 필수)

| 항목 | 가능? | 비고 |
|------|-------|------|
| METI 구독 IAP | ✅ | 표준. 1년 후 수수료 15% |
| 별을 해피트리 **웹**에서 판매 | ✅ | 앱마켓 무관 |
| 웹에서 산 별이 앱/WebView에서 **동작만** | ✅ | "킨들 모델"(크로스플랫폼 소비) |
| 앱 안에 "별 충전 ▸" 버튼·링크·유도 문구 | ❌ | anti-steering 위반 → 리젝 (US·한국·EU 일부만 예외) |
| METI 앱 **인앱 WebView**에서 별 결제 | ❌ | 디지털재화 우회로 간주 → 리젝 위험 |
| METI 포인트를 유저에게 직접 판매 | ❌(지양) | 판매 시 IAP 대상화. **적립 전용 유지** |

> "결제 유도"는 **앱 밖**(해피트리 웹 자체 마케팅·이메일·외부광고)에서만.

## 5. 서버팀(THE-METI) 관점 — 이미 있는 것 / 필요한 것

### 이미 구현됨 (v1.7 §15)
- `POST /partner/users/map` — 사용자 매핑
- `POST /partner/rewards` — 리워드 지급(S2S) → METI 포인트/`reward_balances` 적립
- `GET /partner/user-balance` — 잔액 조회
- `GET /partner/services` — 파트너 목록(+`webview_url`)

### 추가 설계 검토 필요
1. **정산 데이터**: `/partner/rewards` 수신 시 송객/CPA/레브셰어 **정산 카운트**를 별도 적재(파트너별·기간별 집계). 월정산 리포트용.
2. **포인트 일원화 결정**: 현재 `point_wallets`(METI 포인트)와 `reward_balances`(파트너 리워드)가 **분리**돼 있음 → 제휴 적립을 어느 쪽에 쌓을지 통일 필요. (제안: METI 포인트 단일화)
3. **송객 매핑 토큰**: METI 앱 → 해피트리 웹 진입 시 유저 식별 토큰 발급 엔드포인트(예: `POST /partner/:id/launch-token`) 검토.
4. **적립 룰 설정**: 행동별 포인트(출석/게임/별구매 인증 등) 어드민 설정값.

---

## 6. 미결정 사항 (비즈니스)

- 정산 모델: CPI(설치당) vs CPA(별 구매 %당) vs 고정 레브셰어 — 해피트리와 협의
- METI 포인트 적립 단가/한도
- 해피트리 자체 구독 유무 및 위치(웹=Stripe 권장)
- 1차 게임 라인업(출석·룰렛·복권·해피트리·미니게임 N종)
